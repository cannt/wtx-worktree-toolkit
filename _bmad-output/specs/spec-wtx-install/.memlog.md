- (direction) Feature request: unified guided install wizard for wtx, comparable in feel to npx bmad-method install
- (direction) Slug: wtx-install (describes the feature being specced)
- (direction) User-facing entry point: wtx install subcommand (WHAT only; implementation wiring is CA territory)
- (direction) In scope: guided TUI wizard, reference-templating engine, hooks setup, extras menu, idempotency, dry-run mode, completion summary
- (direction) Out of scope: graphify integration, implementation architecture, new config schema fields, CI/CD/packaging, auto-detection of forge type from git remote
- (constraint) bash 3.2 compatible: no readlink -f, no associative arrays, no process substitution to array
- (constraint) set -u only, never set -e in new code
- (constraint) No hardcoded org/project/Jira/forge literals: all values sourced from user input at wizard runtime
- (constraint) Every interactive prompt has a pure-bash read fallback when gum is absent
- (constraint) No eval of raw user input anywhere
- (constraint) All file paths resolved relative to WTX_ROOT or WORKSPACE_ROOT
- (constraint) Config reads via wtx_config_get/wtx_config_get_list only; no ad-hoc TOML parsing in new code
- (constraint) Graceful degradation: absence of gum, jq, claude, or timeout at any wizard step must not abort the flow
- (capability) CAP-1: Guided install wizard — user runs wtx install in any git workspace and is walked through the complete setup via a staged TUI (gum when available, pure-bash read otherwise); success: wizard completes without errors and leaves the workspace fully configured
- (capability) CAP-2: Reference-templating engine — every placeholder in the generated wtx.toml is replaced by a value the user supplies during the wizard; success: the written wtx.toml contains no placeholder values from wtx.example.toml
- (capability) CAP-3: Hooks setup — wizard offers to install Claude Code lifecycle hooks into the current workspace, explaining each hook before confirming; success: hooks files are present in .claude/hooks/ and match the source files in WTX_ROOT/hooks/
- (capability) CAP-4: Extras menu — after core install the wizard offers optional extras (Gradle init script, shell rc PATH hint); success: each offered extra is installed only when the user confirms, and skipped cleanly otherwise
- (capability) CAP-5: Idempotency — when wtx.toml already exists the wizard detects it and offers skip/overwrite/merge before touching any files; success: re-running wtx install never silently overwrites an existing config
- (capability) CAP-6: Dry-run mode — wtx install --dry-run previews every action without writing or modifying any file; success: a dry run produces identical console output to a real run except all file operations are marked [dry-run] and no files change on disk
- (capability) CAP-7: Completion summary — after all steps the wizard shows a styled summary of every action taken and the exact wtx doctor command to verify the install; success: the summary lists each installed item and wtx doctor exits 0 when run after a successful wizard
- (assumption) Assumed: wizard is always invoked from inside a git repository; the flow may check for this and error early if not
- (assumption) Assumed: install.sh remains the canonical authority for symlink creation; the wizard delegates to it rather than reimplementing the symlink logic
- (assumption) Assumed: wtx init remains independently usable as a lower-level config command for scripted/testing use; the wizard does not remove or deprecate it
- (assumption) Assumed: Jira integration is optional at install time; the wizard allows the user to enter zero Jira project key mappings
- (question) Merge behavior (CAP-5): when the user chooses merge, should each wizard prompt be pre-filled with the value from the existing wtx.toml, or should the file be accepted as-is with only missing fields prompted?
- (question) Setup hook discovery: is a directory scan of WTX_ROOT/plugins/ sufficient, or do plugins need a manifest (name, description) for the TUI to display meaningfully?
- (question) PATH check: should the wizard detect whether the install prefix/bin is already on PATH and suppress the PATH hint when it is?
- (decision) UX flow detail (multi-screen wizard steps with gum mockups) is load-bearing and too long for kernel; routing to ux-flow.md companion
- (event) Self-validate Pass 1 (Coherence) — all 7 capabilities have both intent and success; intents describe WHAT not HOW; all 8 constraints rule out design decisions; 1+ explicit non-goals present; success signal is concrete and demonstrable; capability IDs CAP-1..CAP-7 stable and unique; prose is lean — PASS
- (event) Self-validate Pass 2 (Preservation) — all load-bearing source claims mapped: wizard capability->CAP-1, templating engine->CAP-2, hooks->CAP-3, extras->CAP-4, idempotency->CAP-5, dry-run->CAP-6, summary->CAP-7; UX flow detail routed to ux-flow.md companion; bash/set-u/no-eval/gum-fallback/path-resolution constraints all in Constraints section; non-goals cover all out-of-scope items from input; wrapper ceremony dropped — PASS
